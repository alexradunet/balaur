This index is the map for the **object/node generalization program** (plans
164–168): turning Balaur's existing `nodes`/`edges` "knowledge spine" into a
generic, typed, AI-leverageable object system in the spirit of Capacities /
Anytype — *without* a breaking reshape. Each plan is shippable on its own and
written for an executor with **zero context** from the advisory session.

## Program decisions (read before executing any plan)

These hold for **all** plans 164–168. They are deliberate and load-bearing:

1. **Additive, data-preserving incremental migrations only. Do NOT edit the
   consolidated baseline `migrations/1749600000_init.go`.** This checkout runs
   on the owner's **live production VPS** with real data (conversation, tasks,
   journal, life-log). A baseline rewrite only takes effect on a *fresh*
   `pb_data/` and would diverge from — or destroy — the live applied state.
   Every schema change ships as a NEW migration file with a unique,
   strictly-increasing timestamp prefix (`> 1749600000`) registered via
   `m.Register(up, down)`. `tests.NewTestApp` runs the full registered chain
   (today that is only the consolidated baseline — the first incremental file
   this program adds is the first ever exercised that way; verified, not
   assumed), so incremental migrations are exercised by `go test ./...`.
   **Pinned prefixes for this program (strictly increasing, 10-digit, no
   wildcards):** 164 = `1750000000`, 165 = `1750000010`, 167 = `1750000020`,
   168 = `1750000030` (166 adds no migration).
2. **The unified spine stays one collection.** New object capability is `type`
   + `props` + `edges` over the existing `nodes`/`edges` tables, plus a small
   `node_types` *config* registry (sibling to `llm_models`/`owner_settings`,
   not a parallel knowledge store). Do not fragment knowledge across new tables.
3. **The consent boundary is sacred (`PRODUCT.md` pillar).** Agent/model-created
   nodes of a type whose `born_status = proposed` are born proposed and need
   owner approval; traversal AND search stay filtered to `status = active`.
   No plan may let the model write active knowledge without owner consent.
4. **Every mutation is audited after the successful write** (`store.Audit`,
   never before — `AGENTS.md`).
5. **Update `internal/self/knowledge.md` in the same plan** whenever a change
   alters architecture or capability (`AGENTS.md`). A stale self-description
   makes Balaur lie about itself.

## Dependency notes

- **164 is the foundation** — it makes `type` extensible and stands up the
  registry every later plan reads. Nothing else lands until 164 is green.
- **165 requires 164** — property schemas/templates live ON a type definition.
- **166 requires 164** and is sharper after 165 (it can query by property), but
  the link/traverse verbs only need the open type from 164.
- **167 and 168 require 164 + 165** — a task / a measure is registered as a type
  with a property schema, then existing rows are migrated into `nodes`. They are
  independent of each other and can run in either order.

## What this program intentionally still defers

(So nobody re-audits these as gaps. Each remains additive on the same spine.)

- LogSeq-style `((block))` granularity — the node is the unit, not the block.
- A Datalog/SQL-ish query *language* — 166 ships structured query verbs, not a
  query DSL.
- Folding `conversations`/`messages` into nodes — they are an append-heavy
  message log, not objects; they stay relational (and `summaries` with them).
- An interactive/force-directed graph editor — read-only related/traversal only
  (the existing plan 163 view).

## Findings considered and rejected

- *"Make object types meta-circular (type=`object_type` nodes)"* — rejected for
  v1: it would force excluding schema rows from search, traversal, recall, and
  the graph view everywhere (more special-casing than value). A small config
  collection (`node_types`) is simpler and matches how `llm_models` /
  `owner_settings` already coexist with the spine. Revisit only if the owner
  wants type definitions to themselves be linkable/searchable objects.
  **Confirmed against the real systems (web research, 2026-06):** Capacities
  exposes types-as-data for READ only — schema authoring is UI-only (Object
  Studio), not writable via its API; Anytype reifies types/relations as objects
  *because* it is a schema-less, SQL-less, CRDT-synced P2P store with no
  migrations — reification is how it gets introspectable schema at all.
  PocketBase already gives Balaur that legibility (inspectable SQLite +
  migrations + indexes) for free. The chosen hybrid — open `nodes.type` string
  validated against a closed `node_types` row — is exactly Capacities'
  `structureId` / Anytype's `type_key` *interface* without the meta-circular
  *storage* cost. Decision unchanged; evidence strengthened.

## Execution log (2026-06-25)

All 18 plans were executed and reviewed via the `improve` skill's `execute` flow:
each ran in an **isolated git worktree** (a cheaper executor implements + commits;
never directly on `main`), then an **adversarial reviewer** re-ran every done
criterion, checked scope, read the full diff, and audited the tests. The advisor
rendered the final verdict; only APPROVED work was merged to `main`, gated on a
green `go test ./...` + staticcheck, and pushed to `origin`. Run across 5 waves,
serialized so each wave's worktrees branch off `origin/main` with prior waves
present.

The review loop caught — and kept off `main` — each of these before merge:

- **175** — plan infeasible (Go import cycle `life`↔`tasks`: `life/day.go`
  already imports `tasks`). Re-scoped to read `measure` nodes directly in
  `internal/tasks`, then landed.
- **175 / 190 / 183 / 192** — executors ran `graphify update .` inside their
  worktrees, baking ephemeral worktree paths into the tracked
  `graphify-out/graph.json`. Merged **source-only** (the graph is regenerated on
  `main`, not from a throwaway path).
- **183** (3 review rounds) — a chat-hot-path **data race** (`Touch` mutating
  cached records via the by-reference `nodes.Props` map) → a **vacuous** `-race`
  regression test (passed even when the fix was reverted) → a cache that **never
  actually removed the per-turn scan** (computed before the `GetOk` check). Final
  version: `Clone` race-fix + a deterministic throwaway-copy guard + a `GetOk`
  fast-path + a warm-read no-rescan guard; `-race` clean.
- **191** — `node_edit` could mutate `memory`/`skill` nodes, bypassing the
  consent-gated `propose_edit` boundary (a product pillar). Refined to reject any
  non-owner-authored type.
- **184** — a flaky insert-order/timestamp-dependent test (executor STOPPED).
  Refined to a deterministic `StatusRejected`-based test, proven non-flaky at
  `-count=200`.

**Known minor follow-up:** plan 183 introduced a package-level, test-only
`contextCacheComputes atomic.Int64` counter in `internal/knowledge` service code
— a soft brush against the "no global mutable state" rule. Justified (the plan's
behavioral assertion alone could not catch the round-4 bug) and reviewer-accepted,
but a candidate for a cleaner test seam later.

## ✅ RESOLVED (was ⛔ BLOCKER) — date-fragile seed test reded the gate (discovered 2026-06-29, executing 204)

**RESOLVED by plan 211** (`runAt(app, now)` seam; the test now anchors a fixed

Lesson kept: gate every merge with `-count=1` so Go's test cache can't mask a
date-dependent failure. Original diagnosis preserved below for the record.

**`internal/seed` `TestSeedSummariesPopulatesAllBandLevels` failed every Monday**, so a
fresh `go test ./...` is RED — the pre-commit hook + the merge gate block ALL
commits (including 204, and even docs commits). This is a **pre-existing bug,
NOT caused by any 199–210 plan** — it lives in `internal/seed`, outside this batch.

Root cause: the test calls `seed.Run(app)` with the **real `time.Now()`** (its
sibling `TestSeedPeriodsCoversAllBands` uses a *fixed* `2026-06-24`, so it's
deterministic). `recap.Bands` builds the "day" band as *days of the current ISO
week, before today* — on a **Monday** that set is empty by design, so the seed
writes no `day` summary and the test's "≥1 row per period_type" assertion fails
("no day row found"). It passes Tue–Sun, fails Mon.

Why it wasn't caught earlier in this session: **Go's test cache.** Plain
`go test ./...` reused a cached `internal/seed` PASS from a non-Monday run; the
executor's fresh worktree (no cache) and `-count=1` runs exposed the real Monday
failure. Run gates with `-count=1` (or `go clean -testcache`) to avoid masking.

**Recommended fix (deferred, owner to decide — see the held AskUserQuestion):** add
a `runAt(app, now)` seam in `internal/seed` so `Run` keeps real `time.Now()` in
production but the test anchors a fixed non-Monday date (mirroring
`TestSeedPeriodsCoversAllBands`). Small, doctrine-aligned, deterministic.
