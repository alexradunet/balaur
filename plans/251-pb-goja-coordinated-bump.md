## Why this matters

Balaur pins PocketBase at v0.39.3; upstream has shipped v0.39.4 and v0.39.5
(patch-only, no breaking changes, no security content). The catch is a hidden
coupling: PocketBase's own `go.mod` requires
`github.com/dop251/goja v0.0.0-20260618133527-c9b2ea77db59`, so **any**
PocketBase patch bump lets Go's minimum-version-selection (MVS) drag the
sandbox JS engine ~3 months past the commit this repo deliberately reviewed
and pinned (`v0.0.0-20260311135729-065cd970411c`, 2026-03-11). goja runs
untrusted-author extension JS in `internal/ext/vm.go`, and the repo's own
rules (AGENTS.md, quoted below) make any goja version change a gated,
human-reviewed act â€” but no mechanical gate enforces that, so a routine
"bump PocketBase" would smuggle the goja move inside a multi-line dependency
diff and skip the gate. This plan does the two bumps **together, on purpose**:
land PB v0.39.5 + goja `20260618133527` in one change, run the sandbox
regression suite, and attach the goja commit-range list to the report so the
owner can perform the required human review before merge.
