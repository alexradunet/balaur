## Why this matters

Balaur's Chronicle (the recap telescope) is generated by `EnsureSummaries` in
`internal/recap/generate.go`, an hourly idempotent catch-up that summarises
days, then weeks/months/quarters/years, resuming from persisted high-water
marks in `owner_settings`. Two defects make those marks lie:

1. **A silent permanent hole.** The day walk only advances its mark past a day
   when a summary was actually written (or the day genuinely had no chat). The
   parent walk (week/month/quarter/year) has NO such guard: it marks every
   fully-past period as done regardless of outcome. `ensureOne` returns
   `(false, nil)` when the model emits whitespace-only text — so during a
   multi-period catch-up (initial history walk, downtime longer than one
   period), an empty generation for an EARLIER period is skipped forever once a
   LATER period settles in the same run. The next run resumes at the mark and
   never revisits the failed period. The owner's Chronicle gets a permanent
   missing week/month with no error anywhere.
2. **Timezone change orphans all history.** Summaries are keyed by
   `period_start` computed in the owner's zone (local midnight → UTC). If the
   owner changes the `timezone` owner-setting, every historical period computes
   a DIFFERENT `period_start`, so lookups miss all old summaries — but the
   marks still claim "done through X", so nothing before the mark is ever
   regenerated. Before the high-water marks existed, the full re-walk
   self-healed this.

After this plan: the parent walk only advances its mark through contiguously
settled periods (a failed generation is retried on the next run), and each mark
is stamped with the zone it was computed in — a zone change invalidates the
marks and triggers exactly one full re-walk under the new zone (cheap when
summaries exist: the exists-check short-circuits without model calls).

### The high-water helpers as they exist today

`internal/recap/generate.go:81-97` (key builders — note the doc comment
documents the value format, which this plan changes):

```go
// highWaterKey is the owner_settings key holding the newest CONTIGUOUSLY
// summarised day for one conversation (value: "YYYY-MM-DD" in the owner's
// wall clock). It lets the hourly catch-up resume past already-settled days
// instead of re-walking all history; the Find/exists short-circuit in
// ensureOne remains the correctness safety net, so a stale mark can never
// PERMANENTLY skip a genuinely-missing summary.
func highWaterKey(conversationID string) string {
	return "recap_highwater_" + conversationID
}

// parentHighWaterKey is the owner_settings key for the newest fully-past
// period of the given type (week/month/quarter/year) that has been walked for
// one conversation. Mirrors the day high-water pattern, with one mark per
// period type so week/month/quarter/year resume independently.
func parentHighWaterKey(conversationID, periodType string) string {
	return "recap_parent_highwater_" + conversationID + "_" + periodType
}
```

`internal/recap/generate.go:99-122`:

```go
// loadHighWater returns the persisted high-water date stored under key as a
// local-midnight time in loc, or the zero time when none is stored or it is
// unparseable (so the caller falls back to walking from oldest).
func loadHighWater(app core.App, key string, loc *time.Location) time.Time {
	raw := store.GetOwnerSetting(app, key, "")
	if raw == "" {
		return time.Time{}
	}
	t, err := time.ParseInLocation("2006-01-02", raw, loc)
	if err != nil {
		return time.Time{} // unreadable mark → fall back to oldest, never crash
	}
	return t
}

// saveHighWater records date d (local midnight) under key. Best-effort: a
// failure to persist only means the next run re-walks from the previous mark —
// the Find short-circuit keeps that correct, just not maximally cheap. Never
// abort the run on this.
func saveHighWater(app core.App, key string, d time.Time) {
	if err := store.SetOwnerSetting(app, key, d.Format("2006-01-02")); err != nil {
		app.Logger().Warn("recap: high-water persist failed", "error", err)
	}
}
```

### ensureOne — the three meanings of `(false, nil)`

`internal/recap/generate.go:233-275`:

```go
// ensureOne generates and stores one period summary if missing and its
// period is complete. Reports whether a summary now exists.
func ensureOne(ctx context.Context, app core.App, client llm.Client, conversationID string, p Period, now time.Time) (bool, error) {
	if p.End.After(now) {
		return false, nil // period still running
	}
	if Find(app, conversationID, p) != nil {
		return true, nil // already done — idempotency
	}

	var source string
	var count int
	var err error
	if p.Type == "day" {
		source, count, err = daySource(app, conversationID, p)
	} else {
		source, count, err = childSource(app, conversationID, p)
	}
	if err != nil {
		return false, err
	}
	if source == "" {
		return false, nil // silence is not an error; quiet days leave no card
	}

	stream, err := client.ChatStream(ctx, summarisePrompt(p, source), nil)
	if err != nil {
		return false, fmt.Errorf("summarising %s: %w", periodLabel(p), err)
	}
	text, err := llm.Collect(stream)
	if err != nil {
		return false, fmt.Errorf("summarising %s: %w", periodLabel(p), err)
	}
	if strings.TrimSpace(text) == "" {
		return false, nil
	}
	if err := save(app, conversationID, p, strings.TrimSpace(text), count); err != nil {
		return false, err
	}
	store.Audit(app, "recap", "recap.generate", p.Type+"/"+p.Start.Format("2006-01-02"), true,
		map[string]any{"sources": count})
	return true, nil
}
```

So `(false, nil)` conflates: (1) period still open, (2) quiet period (no
source), (3) whitespace-only generation. The day walk distinguishes (2) from
(3) by re-calling `daySource`; the parent walk doesn't distinguish anything.

### The day walk (has the guard) vs the parent walk (missing it)

Day walk, `internal/recap/generate.go:294-323`:

```go
	start := oldest
	if hw := loadHighWater(app, highWaterKey(conversationID), now.Location()); hw.After(start) {
		start = hw
	}
	contiguous := time.Time{}
	stillContiguous := true
	for d := Day(start); d.End.Before(now) || d.End.Equal(now); d = Containing("day", d.End) {
		ok, err := ensureOne(ctx, app, client, conversationID, d, now)
		if err != nil {
			return err

			settled := ok

				// quiet (no source), not if a summary failed to materialise.

			} else {
				stillContiguous = false // stop advancing the mark past this gap
			}
		}
	}
	if !contiguous.IsZero() {
		saveHighWater(app, highWaterKey(conversationID), contiguous)
```

### One behavior that is DELIBERATE and must survive unchanged

`TestEnsureSummariesParentHighWaterSkipsPast`
(`internal/recap/generate_test.go:364-402`): a summary the owner DELETED for a
period strictly below an already-persisted mark is NOT regenerated. The mark
is a performance cut, not a correctness guarantee for pre-mark periods. Your
new guard must not break this — and it doesn't, structurally: the guard only
controls how far the mark ADVANCES during the current run; a deletion below an
already-advanced mark is never re-visited because the walk starts AT the mark.
In that test the rerun starts at the March mark, finds March existing
(`ensureDone`), finds April–June quiet (`ensureQuiet`), and never reaches
January. Zero model calls, Jan stays gone — the test stays green. (A deleted
pre-mark summary is indistinguishable from one that never generated; accepting
that is the documented trade-off of the mark.)

### Step 1: Give ensureOne an honest result type

In `internal/recap/generate.go`, replace `ensureOne`'s `(bool, error)` return
with a small result enum so callers can distinguish the three `(false, nil)`
meanings. Immediately above `ensureOne` (currently `generate.go:233`), add:

```go
// ensureResult classifies what ensureOne left behind for a period, so the
// high-water walks can tell "retry next run" apart from "settled forever".
type ensureResult int

const (
	ensureOpen  ensureResult = iota // period still running — re-evaluate next run
	ensureDone                      // a summary exists (pre-existing or just written)
	ensureQuiet                     // period complete but has no source — nothing will ever appear
	ensureEmpty                     // generation produced only whitespace — retry next run
)
```

Change the signature to:

```go
func ensureOne(ctx context.Context, app core.App, client llm.Client, conversationID string, p Period, now time.Time) (ensureResult, error)
```

and map the existing return statements exactly:

- `p.End.After(now)` → `return ensureOpen, nil`
- `Find(...) != nil` → `return ensureDone, nil`
- source-load error → `return ensureEmpty, err` (a non-settled value, so even a
  caller that forgot to check err could not advance a mark)
- `source == ""` → `return ensureQuiet, nil`
- `ChatStream` / `Collect` errors → `return ensureEmpty, fmt.Errorf("summarising %s: %w", periodLabel(p), err)` (keep the existing wrapping)
- `strings.TrimSpace(text) == ""` → `return ensureEmpty, nil`
- save error → `return ensureEmpty, err`
- success (after the `store.Audit` call, which stays exactly where it is —
  strictly after the successful save) → `return ensureDone, nil`

Update the doc comment on `ensureOne` ("Reports whether a summary now
exists" → describe the result classification in one sentence).

## Maintenance notes

- **Orphaned rows after a zone change are accepted, not cleaned.** The re-walk
  regenerates summaries under the new zone's period starts; the old-zone rows
  stay in `summaries`, unreferenced by any lookup. They are derived data
  (regenerable from messages) and harmless; a cleanup sweep is deliberately out
  of scope. If a dedupe ever lands, key it on (conversation, period_type,
  period label under the CURRENT zone), not raw period_start.
- **`time.Local` stamps as `"Local"`.** When the `timezone` owner-setting is
  unset, `store.OwnerLocation` returns `time.Local` (`internal/store/time.go:22-32`)
  and the mark is stamped `|Local`. A change of the HOST system zone without
  setting the owner-setting is therefore NOT detected (the stamp stays
  `"Local"`). That matches the pre-existing `OwnerLocation` semantics; fixing
  it would require stamping the resolved offset or name behind `time.Local`,
  which is out of scope.
- **One-time re-walk on upgrade.** Every existing box has legacy zoneless marks
  (`YYYY-MM-DD`); the first `EnsureSummaries` run after this lands re-walks all
  history once (a `Find` per period, zero model calls where summaries exist)
  and re-persists stamped marks. Reviewers should expect one slow-ish cron tick
  on boxes with years of history, not a behavioral bug.
- **Regeneration of deleted pre-mark summaries after a zone change** is a side
  effect of invalidating the marks: an owner who deleted a summary and later
  changes timezone will see a fresh summary for that period under the new zone.
  That is consistent with "the mark is a performance cut, not a deletion
  ledger" (see `TestEnsureSummariesParentHighWaterSkipsPast`'s doc comment).
- **What a reviewer should scrutinize:** the parent-walk guard must not stop
  the LOOP, only the MARK (later periods still generate in the same run); the
  `store.Audit` call in `ensureOne` must remain strictly after `save`; and the
  `|` format must round-trip through `strings.Cut` for zone names containing
  `/` (IANA names do; they never contain `|`).
- Deferred: a `week`/`month` mark could in principle also be invalidated when
  the owner's locale changes week conventions — Balaur only does ISO weeks
  (`internal/recap/periods.go:23-31`), so this is moot until locale-aware weeks
  exist.
